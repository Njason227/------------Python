# Техническое задание
## Программа «Распределение психиатрических пациентов»
### Приёмное отделение психиатрической клиники

---

## 1. Общие сведения

**Название:** Распределение психиатрических пациентов  
**Назначение:** Автоматизация процесса распределения пациентов, поступающих в приёмное отделение психиатрической клиники, по профильным отделениям на основании клинических критериев  
**Платформа:** Веб-приложение  
**Стек технологий:** Python 3.12+, Django 5.x, PostgreSQL 16+, HTML5/CSS3/JS  
**Адаптивность:** Веб-интерфейс обязан корректно отображаться на устройствах с разным разрешением экрана (десктоп, планшет, мобильные устройства)

---

## 2. Функциональные требования

### 2.1. Справочники

| Справочник | Описание |
|---|---|
| **Пол** | Мужской, Женский |
| **Возрастная категория** | До 18 лет (детский/подростковый), 18 лет и старше (взрослый) |
| **Диагнозы по МКБ-11** | Классификация психических расстройств по Международной классификации болезней 11-го пересмотра (рубрики F00–F99) с иерархической структурой: класс → блок → категория → конкретный диагноз |
| **Тяжесть психического состояния** | Лёгкая, Умеренная, Тяжёла, Крайне тяжёлая |
| **Тяжесть физического состояния** | Удовлетворительное, Средней тяжести, Тяжёлое, Критическое |

> Справочник диагнозов МКБ-11 должен быть внесён в базу данных и доступен для выбора через интерфейс. Допускается начальное заполнение из CSV/JSON-файла.

### 2.2. Управление пациентами (CRUD)

Реализовать полный цикл операций для сущности **Пациент**:

- **Создание** — регистрация нового пациента в приёмном отделении
- **Просмотр** — детальная карточка пациента со всеми атрибутами
- **Редактирование** — изменение данных пациента
- **Удаление** (мягкое удаление / деактивация)

**Обязательные поля пациента:**

| Поле | Тип данных | Описание |
|---|---|---|
| `id` | UUID / автоинкремент | Уникальный идентификатор |
| `ФИО` | строка (.CharField) | Фамилия, имя, отчество |
| `Дата рождения` | дата | Используется для автоматического расчёта возраста и определения возрастной категории |
| `Пол` | ссылка на справочник | Мужской / Женский |
| `Диагноз (МКБ-11)` | ссылка на справочник | Основной диагноз |
| `Тяжесть психического состояния` | ссылка на справочник | Степень тяжести |
| `Тяжесть физического состояния` | ссылка на справочник | Степень тяжести |
| `Дата и время поступления` | дата+время | Автоматически при создании |
| `Статус` | строка | В ожидании / Распределён / Госпитализирован / Отказ |
| `Назначенное отделение` | ссылка на отделение | Заполняется при распределении |

### 2.3. Логика распределения

Реализовать алгоритм автоматического распределения пациента в профильное отделение на основании совокупности параметров:

**Правила распределения (примерные, подлежат уточнению с заказчиком):**

1. **По полу и возрасту** — определение блока (детское/подростковое, мужское взрослое, женское взрослое)
2. **По диагнозу МКБ-11** — привязка к профильному отделению (например: F20–F29 → отделение шизофрении и расстройств шизофренического спектра, F30–F39 → аффективные расстройства и т.д.)
3. **По тяжести психического состояния** — приоритетность и тип палаты (интенсивная терапия, общее отделение, обсервационное)
4. **По тяжести физического состояния** — необходимость совместного наблюдения с соматическим отделением или усиленного медицинского контроля

**Функциональные возможности:**

- Автоматическое распределение по нажатию кнопки (одиночное и массовое)
- Отображение рекомендованного отделения с обоснованием
- Возможность ручного переназначения отделения врачом с указанием причины
- Логирование каждого решения о распределении (кто, когда, куда, по каким критериям)

### 2.4. Просмотр распределений

- Таблица текущих распределений с фильтрацией по отделению, дате, статусу
- Карточка распределения с полной историей изменений
- Список ожидающих распределения (очередь)

### 2.5. Отчётность и статистика

- Количество пациентов по отделениям (текущее)
- Загрузка отделений (от общего количества коек)
- Распределение по полу, возрастной категории, диагнозу, тяжести
- Динамика поступлений за период (день, неделя, месяц)
- Экспорт отчётов в CSV / PDF

### 2.6. Учётные отделения клиники (справочник)

| Поле | Описание |
|---|---|
| Название отделения | Например: «Отделение шизофрении» |
| Профиль | Основной тип диагнозов |
| Пол | Мужское / Женское / Смешанное |
| Возрастная категория | Детское / Взрослое |
| Количество коек | Общее число мест |
| Занято коек | Текущая загрузка |

## 3. Нефункциональные требования

### 3.1. База данных

- **СУБД:** PostgreSQL 16+
- **ORM:** Django ORM
- **Миграции:** Управление через `python manage.py migrate`
- **Начальное заполнение:** Django fixtures или management-команда для загрузки справочника МКБ-11 и отделений

### 3.2. Веб-интерфейс

- **Адаптивная вёрстка:** CSS-фреймворк (Bootstrap 5 или Tailwind CSS)
- **Респонсивность:** Корректное отображение при ширине от 320px (мобильные) до 2560px (десктоп)
- **Навигация:** Верхнее меню / боковая панель (сворачиваемая на мобильных)
- **Формы:** Валидация на стороне клиента и сервера
- **Таблицы:** Пагинация, сортировка, фильтрация

### 3.3. Аутентификация и авторизация

- Встроенная система аутентификации Django
- Минимально две роли:
  - **Врач приёмного** — регистрация и редактирование пациентов, распределение
  - **Администратор** — управление справочниками, отделениями, просмотр статистики
- При необходимости: роль «Старшая медсестра» — просмотр, ограниченное редактирование

### 3.4. Производительность

- Время загрузки основных страниц — не более 2 секунд
- Поддержка одновременной работы минимум 10 пользователей

### 3.5. Безопасность

- Защита от CSRF, XSS
- HTTPS (рекомендация для промышленной эксплуатации)
- Хеширование паролей (встроено в Django)
- Журнал аудита действий пользователей

## 5. Основные модели данных

### Модель «Пациент» (Patient)

```
Patient
├── id               — AutoField / UUIDField (PK)
├── last_name        — CharField(max_length=100)
├── first_name       — CharField(max_length=100)
├── patronymic       — CharField(max_length=100, blank=True)
├── date_of_birth    — DateField
├── gender           — ForeignKey → Gender
├── diagnosis        — ForeignKey → Diagnosis (МКБ-11)
├── mental_severity  — ForeignKey → MentalSeverity
├── physical_severity— ForeignKey → PhysicalSeverity
├── admission_date   — DateTimeField(auto_now_add=True)
├── status           — CharField(choices=STATUS_CHOICES)
├── department       — ForeignKey → Department (nullable)
├── assigned_by      — ForeignKey → User (nullable)
├── assignment_note  — TextField(blank=True)
└── created_at / updated_at — DateTimeField
```

### Модель «Отделение» (Department)

```
Department
├── id                — AutoField (PK)
├── name              — CharField(max_length=200)
├── profile           — CharField / ForeignKey (профиль диагнозов)
├── gender_restriction — CharField(choices: M/F/ANY)
├── age_category      — CharField(choices: CHILD/ADULT/ANY)
├── total_beds        — PositiveIntegerField
├── occupied_beds     — PositiveIntegerField(default=0)
└── is_active         — BooleanField(default=True)
```

### Модель «Диагноз МКБ-11» (Diagnosis)

```
Diagnosis
├── id           — AutoField (PK)
├── code         — CharField(max_length=10)   # например: F20.0
├── name         — CharField(max_length=500)
├── block        — CharField(max_length=200)  # Блок МКБ
├── chapter      — CharField(max_length=200)  # Класс МКБ
└── parent       — ForeignKey → self (nullable, для иерархии)
```

### Модель «Лог распределения» (AssignmentLog)

```
AssignmentLog
├── id               — AutoField (PK)
├── patient          — ForeignKey → Patient
├── from_department  — ForeignKey → Department (nullable)
├── to_department    — ForeignKey → Department
├── assigned_by      — ForeignKey → User
├── reason           — TextField
├── is_automatic     — BooleanField
└── created_at       — DateTimeField(auto_now_add=True)
```

### Вспомогательные модели-справочники

- **Gender** — id, name (Мужской, Женский)
- **MentalSeverity** — id, name, level (1–4)
- **PhysicalSeverity** — id, name, level (1–4)

При необходимости — REST API (Django REST Framework):

## 8. Требования к развёртыванию

- Python 3.12+
- Django 5.x
- PostgreSQL 16+
- pip + virtualenv (или poetry)
- Файл `requirements.txt` с фиксацией версий
- Переменные окружения: `DJANGO_SECRET_KEY`, `DATABASE_URL`, `DEBUG`
- Команды:
  - `python manage.py migrate` — применение миграций
  - `python manage.py loaddata fixtures/*.json` — загрузка справочников
  - `python manage.py createsuperuser` — создание администратора
  - `python manage.py runserver` — запуск сервера разработки
